PROJECT OVERVIEW
================================================================================
A dense retrieval benchmark system for HotpotQA with GPU acceleration and 
multiple sampling strategies. Evaluates embedding models on document retrieval 
tasks using FAISS indexing.

Key Stats:
  - 2,720 lines of Python code across 12 files
  - Main framework: PyTorch + Transformers + FAISS
  - Supports 8+ embedding models (BGE, E5, GTE)
  - 3 sampling strategies (top-k, random, diverse)
  - 5 evaluation metrics (Pass@k, Recall@k, Precision@k, NDCG@k, MRR)

================================================================================
CORE MODULES
================================================================================

1. CONFIGURATION (config.py - 63 lines)
   - BenchmarkConfig: Central dataclass for all settings
   - Supported models: BGE, E5, GTE families
   - Dataset: HotpotQA (fullwiki or distractor modes)
   - Sampling strategies: top_k, random_sample, diverse_sample

2. DATA LOADING (data_loader.py - 368 lines)
   - HotpotQADataLoader: Load from HuggingFace Datasets
   - HotpotQAExample: Individual question with context
   - Paragraph: Sentence-level document unit
   - Ground truth label creation
   - Wikipedia corpus support (21M passages)

3. RETRIEVAL MODELS (2 files)
   a) retrieval_model.py (186 lines) - Basic version
      - Simple batch processing
      - No multiprocessing
      - Good for understanding the flow
   
   b) retrieval_model_optimized.py (305 lines) - GPU-optimized
      - DataLoader with multiprocessing
      - CPU tokenization (workers), GPU forward pass
      - Pin memory for faster transfers
      - cuDNN autotuner enabled
      - Auto batch size detection
      - ~2-3x faster than basic version

4. INDEXING (indexer.py - 303 lines)
   - FAISSIndexer: FAISS-based efficient indexing
     * 3 index types: Flat (exact), IVF (approx), IVFPQ (memory-efficient)
     * GPU support with CPU fallback
     * Save/load functionality
     * Normalization handling
   - SimpleRetriever: Numpy-based fallback for small corpus

5. SAMPLING MECHANISMS (sampling_strategies.py - 281 lines)
   - RetrievalSampler: Apply 3 strategies
   
   a) Top-K (Standard)
      - Simple: return top-k documents
      - No sampling overhead
      
   b) Random Sampling
      - Retrieve: top (expansion_factor * k)
      - Sample: random_ratio fraction
      - Re-rank: by score, take top-k
      - Effect: Adds diversity, may hurt rank quality
      
   c) Diverse Sampling
      - Retrieve: top (expansion_factor * k)
      - Stratify: divide into 4 bins
      - Sample: from each bin
      - Re-rank: by score, take top-k
      - Effect: Guaranteed diversity across ranks

6. METRICS (metrics.py - 299 lines)
   - RetrievalMetrics: Static metric calculators
     * Pass@k: binary success metric
     * Recall@k: fraction of relevant found
     * Precision@k: precision in top-k
     * NDCG@k: discounted cumulative gain
     * MRR: mean reciprocal rank
   - BenchmarkEvaluator: Aggregate and format results

7. MAIN PIPELINE (benchmark.py - 415 lines)
   - RetrievalBenchmark: Orchestrates 5-step pipeline
     1. Load dataset from HuggingFace
     2. Build corpus from examples
     3. Encode & index documents
     4. Retrieve for queries
     5. Evaluate against ground truth
   - Command-line interface with 20+ arguments
   - Results saved to JSON with timestamp

================================================================================
5-STEP PIPELINE
================================================================================

STEP 1: LOAD DATA
  Input:  Command-line config
  Action: Load HotpotQA from HuggingFace
  Output: List[HotpotQAExample] (~100 questions for test)
  Time:   1-2 seconds

STEP 2: BUILD CORPUS
  Input:  List[HotpotQAExample]
  Action: Extract sentences, create Paragraph objects
  Output: List[Paragraph] (~5-20 per question)
  Time:   <1 second

STEP 3: BUILD INDEX
  Input:  List[Paragraph]
  Action: Encode with DenseRetrieverOptimized, create FAISS index
  Output: FAISSIndexer ready for search
  Time:   10-20 seconds for 10K docs

STEP 4: RETRIEVE
  Input:  Query questions
  Action: Encode queries, search FAISS, apply sampling
  Output: Dict[question_id -> List[doc_ids]]
  Time:   <1 second for 100 queries

STEP 5: EVALUATE
  Input:  Retrieval results + ground truth
  Action: Calculate 5 metrics at k=[1,3,5,10,20]
  Output: Dict[metric_name -> score]
  Time:   <1 second

================================================================================
SAMPLING STRATEGIES EXPLAINED
================================================================================

Example: k=5 documents, 20 initial results, expansion_factor=4

TOP-K (Standard):
  1. Search returns top 20 documents
  2. Select top 5
  3. Result: [doc_0, doc_1, doc_2, doc_3, doc_4]

RANDOM SAMPLING:
  1. Search returns top 20 documents
  2. Randomly sample 6 documents (20 * 0.3 ratio)
  3. Re-rank by score, take top 5
  4. Result: 5 from random 6 (less correlated with exact rank)

DIVERSE SAMPLING (Stratified):
  1. Search returns top 20 documents
  2. Split into 4 bins (5 docs each)
     - Bin 0: docs 0-4 (best scores)
     - Bin 1: docs 5-9
     - Bin 2: docs 10-14
     - Bin 3: docs 15-19 (worst scores)
  3. Sample from each bin (proportionally)
  4. Re-rank, take top 5
  5. Result: 5 docs from different quality ranges

Key Difference:
  - Top-k: Biased toward high-scoring documents
  - Random: Adds noise, may hurt rank quality
  - Diverse: Balanced coverage of score ranges

MODELS & DATASETS
================================================================================

EMBEDDING MODELS (8 total):

BGE (Best General Embeddings):
  - bge-small-en-v1.5 (33M params)
  - bge-base-en-v1.5 (139M params)
  - bge-large-en-v1.5 (335M params)
  - Optimized for ranking, high quality

E5 (Multilingual E5):
  - e5-small-v2, e5-base-v2, e5-large-v2
  - Requires "query:" and "passage:" prefixes
  - Handles automatically by code

GTE (General Text Embeddings):
  - gte-base, gte-large
  - High quality, multilingual

DATASET: HotpotQA

Structure:
  - Multi-hop QA requiring 2+ documents
  - ~100K training questions
  - Supporting facts annotations (2-3 per question)

Modes:
  - fullwiki: Full Wikipedia (21M passages), harder
  - distractor: 2 relevant + 8 distractors, easier

Splits:
  - train: ~88K questions
  - validation: ~5.9K questions (dev set)
  - test: ~5.5K questions (no supporting facts labels)

Ground Truth:
  - Supporting facts = relevant documents
  - Matched by (title, sentence_id)

================================================================================
GPU ACCELERATION TECHNIQUES
================================================================================

1. DATALOADER MULTIPROCESSING
   - Workers on CPU: Tokenization is I/O bound
   - Main process on GPU: Forward pass is compute bound
   - Result: GPU never idle, CPU never saturated
   - Speedup: ~1.5-2x

2. PINNED MEMORY (pin_memory=True)
   - Allocate CPU memory for faster GPU transfer
   - DMA (Direct Memory Access) to GPU
   - Result: Faster CPU -> GPU data movement
   - Speedup: ~1.2x

3. PERSISTENT WORKERS
   - Keep worker processes alive between batches
   - No process creation/destruction overhead
   - Result: Smooth pipeline
   - Speedup: ~1.1x

4. CUDNN AUTOTUNER
   - Test different algorithms, pick fastest
   - One-time cost per batch size
   - Result: Optimal algorithm selection
   - Speedup: ~1.1x

5. AUTO BATCH SIZE DETECTION
   - Test increasing batch sizes until OOM
   - Use 80% of max for safety
   - Result: Maximize GPU utilization
   - Speedup: ~1.5-3x depending on initial batch size

Combined Effect: ~2-3x faster encoding

================================================================================
KEY INSIGHTS
================================================================================

1. Corpus encoding is the bottleneck (O(N) where N = num_docs)
2. Query encoding is fast (O(M) where M = num_queries, usually M << N)
3. FAISS search is very fast (~1ms per query for 100K docs)
4. Sampling has minimal computational cost (<1% overhead)
5. Evaluation is negligible compared to encoding
6. GPU acceleration gives 2-3x speedup with proper settings
7. Diverse sampling helps generalization but costs retrieval quality
8. Model selection matters more than sampling strategy
